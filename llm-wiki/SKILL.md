# LLM Wiki

基于 [Karpathy 的 LLM Wiki 模式](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)，让 LLM 增量构建并维护一个持久化的 wiki——一组结构化、互相链接的 Markdown 文件，介于你和原始资料之间。

**核心理念**：知识编译一次、持续更新，而非每次查询时重新推导。你负责策划资料来源、引导分析、提出好问题；LLM 负责所有繁琐工作——摘要、交叉引用、归档和记账。

## 适用场景

- **个人成长**：目标、健康、心理、自我提升——积累日志和文章，逐步构建结构化的自我认知
- **深度研究**：数周/数月持续阅读论文、报告，构建带演进论点的综合 wiki
- **读书伴侣**：逐章摄入，构建角色、主题、情节线索页面及其关联
- **团队知识库**：由 Slack、会议记录、项目文档等喂入，LLM 负责维护更新
- **其他**：竞品分析、尽职调查、旅行规划、课程笔记、爱好研究……

任何需要**持续积累和结构化组织**知识的领域都适用。

## 三层架构

| 层 | 说明 | 谁维护 |
|---|------|--------|
| **Raw sources** | 你策划的源文档集合（文章、论文、图片、数据）。不可变——LLM 只读不改。事实来源。 | 用户 |
| **Wiki** | LLM 生成的 Markdown 文件目录（摘要、实体页、概念页、对比、综述）。LLM 创建页面、更新内容、维护交叉引用。你阅读它；LLM 书写它。 | LLM |
| **Schema** | 告诉 LLM wiki 如何组织、遵循什么约定、执行什么工作流的配置文件。把 LLM 从通用聊天机器人变成纪律严明的 wiki 维护者。 | 用户与 LLM 共同演进 |

Schema 文件名随平台而定：`CLAUDE.md`（Claude Code）、`AGENTS.md`（Codex/OpenCode）、`.cursorrules`（Cursor）等。

## 操作

### Ingest（摄入）

你把新资料放入原始资料集合，告诉 LLM 去处理。示例流程：LLM 阅读资料，与你讨论关键发现，在 wiki 中写一个摘要页，更新 index，更新 wiki 中相关的实体页和概念页，在 log 中追加一条记录。一篇资料可能触及 10-15 个 wiki 页面。

建议逐篇摄入、保持参与——阅读摘要、检查更新、引导 LLM 强调什么。也可批量摄入减少监督。由你决定适合自己的工作流，并在 schema 中记录以供后续会话使用。

### Query（查询）

你针对 wiki 提问。LLM 搜索相关页面、阅读它们、综合出带引用的答案。答案形式取决于问题——Markdown 页面、对比表格、Marp 幻灯片、matplotlib 图表、canvas。

**重要洞察**：好的答案可以作为新页面归档回 wiki。一次对比分析、一个发现、一个联系——这些有价值的内容不应该消失在聊天记录中。这样你的探索也像摄入的资料一样在知识库中复利增长。

### Lint（巡检）

定期让 LLM 对 wiki 做健康检查：

- 页面间矛盾
- 被新资料取代的过时内容
- 无入站链接的孤立页面
- 被多次提及但缺少独立页面的重要概念
- 缺失的交叉引用
- 可通过网络搜索填补的数据缺口

LLM 擅长**主动建议**值得探究的新问题和值得寻找的新资料来源。这能在 wiki 增长时保持其健康。

## 索引与日志

两个特殊文件帮助 LLM（和你）在 wiki 增长时导航：

| 文件 | 面向 | 说明 |
|------|------|------|
| **index.md** | 内容 | wiki 所有页面的目录（链接 + 一句话摘要 + 可选元数据），按分类组织。LLM 每次摄入后更新；查询时先读 index 定位再深入。中等规模（~100 篇资料）下无需 RAG。 |
| **log.md** | 时间线 | Append-only 记录。每条以 `## [YYYY-MM-DD] 操作 \| 标题` 开头，可用 `grep "^## \[" log.md \| tail -5` 解析。帮 LLM 了解最近做了什么。 |

## 可选：CLI 工具

当 wiki 规模增长后，你可能需要小工具帮助 LLM 更高效地操作。最明显的是搜索引擎——小规模下 index 文件够用，但规模大了需要正式搜索。[qmd](https://github.com/tobi/qmd) 是一个好选择：本地 Markdown 搜索引擎，支持混合 BM25/向量搜索和 LLM 重排序，全部在本地运行，有 CLI 和 MCP server。也可以自己搭建更简单的方案——LLM 可以帮你写一个朴素的搜索脚本。

## Tips

| 工具 | 用途 |
|------|------|
| Obsidian Web Clipper | 浏览器扩展，网页文章→Markdown，快速收集资料 |
| 图片本地化 | Obsidian 设置附件目录为 `raw/assets/`，绑定快捷键一键下载 |
| Obsidian 图谱视图 | 查看 wiki 形状——枢纽页、孤岛页一目了然 |
| Marp | Obsidian 插件，从 wiki 内容直接生成幻灯片 |
| Dataview | Obsidian 插件，对 YAML frontmatter 运行查询生成动态表格 |
| Git | wiki 就是 markdown 的 git 仓库——版本历史、分支、协作免费获得 |

## 灵活性

本文档有意保持抽象。具体的目录结构、schema 约定、页面格式、工具链——都取决于你的领域、偏好和选择的 LLM。上述提到的一切都是**可选和模块化的**——选取有用的，忽略不需要的。正确的使用方式是把这份文档分享给你的 LLM agent，一起协作出适合你需求的版本。

## 初始化

首次使用时，与 LLM 讨论：

1. 确认 wiki 主题和存放路径
2. 一起确定目录结构（参考 [schema-template.md](schema-template.md) 中的默认结构）
3. 从模板生成 Schema 文件，根据主题调整
4. 创建 `index.md` 和 `log.md`
5. `git init` 并做首次提交
